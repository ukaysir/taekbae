## 1. 확정 아이디어

### 제안명(현재 작업명)

**대전 트램 공사 물류위험 관측·예보 프로토타입 — 도시변화 리스크 모듈**

### 한 문장 정의

대전시·트램 공사 시행기관의 **교통대책 담당자**가 공사 통제계획과 실시간 교통을 결합해 시간대·구간별 물류통행 위험을 비교하고, 그 결과를 기존 운송관리시스템(TMS)과 도심배송 배차담당자에게 위험도·예상 지연시간 형태로 제공하는 의사결정 지원 서비스다.

### 이번 계획에서 고정한 사항

- 트램을 포기하지 않고 **첫 실증사례**로 사용한다.
- 트램 공사가 끝나면 트램 전용 화면은 종료하거나 개통 후 평가화면으로 전환한다.
- 공사에서 학습한 데이터·모델·API 구조는 다른 계획공사와 도로공간 변화에 재사용한다.
- 기존 TMS, 택배 코어시스템, 기사 앱을 대체하지 않는다.
- 현재 검증된 핵심 실증은 **트램 공사구간의 현재 교통 관측과 계획경로 위험정보 결합**이다.
- 30분 후 예측은 구현된 평가 파이프라인이 최소 288스냅숏·48시간·3개 날짜·5,000예제를 충족하고 기준모델을 이길 때만 활성화한다.
- 공사계획 A/B의 인과적 효과 비교는 데이터 조건을 통과할 때만 제시한다. 조건이 부족하면 장기 확장기능으로 명시한다.
- 공사단계 A/B 자료가 부족하므로 ‘시뮬레이터’ 표현을 이미 제거했다.
- 실제 택배사 주문·차량·기사 데이터를 보유하거나 기관과 도입 협의를 마친 것처럼 쓰지 않는다.
- 실제 배송시간·비용·탄소 절감효과는 검증 전 수치로 주장하지 않는다.

## 2. 서비스 사용자와 역할

공모전 제안에서 한 사용자가 모든 일을 하는 것처럼 쓰지 않고, 의사결정자·수혜자·연동대상을 분리한다.

| 역할 | 제안상 대상 | 바꾸려는 의사결정 | 현재 확인 상태 |
|---|---|---|---|
| 1차 의사결정자 | 대전시 또는 트램 공사 시행기관의 교통대책·공정계획 담당자 | 통제 시간대, 단계별 공사계획, 우선관리 구간, 물류업계 사전안내 | **제안 대상**이며 실제 도입·협의는 미확정 |
| 운영 수혜자 | 택배 영업소·도심배송업체 배차담당자 | 공사구간 통과시간, 운행 버퍼, 배송순서 검토 | 수요·사용성 검증 필요 |
| 연동 대상 | 기존 TMS·배차솔루션 운영사 | 기존 배차계획에 링크·시간대별 위험 페널티 반영 | 공개 연동규격과 제휴 가능성 미확인 |
| 간접 수혜자 | 배송기사 | 기존 업무도구 안에서 조정된 운행계획 확인 | 별도 앱을 새로 설치시키지 않는 방향 |

### 대표 사용 장면

1. 공사·교통대책 담당자가 공사구간, 통제기간, 시간대, 통제내용을 등록한다.
2. 시스템은 해당 계획을 교통링크와 연결하고 30분 후 통행시간과 배송노출 위험을 계산한다.
3. 담당자는 시간대별 위험지도와 우선관리 구간을 확인한다.
4. 운영 중에는 예측과 실제 교통을 비교해 공사 안내와 통제계획 검토에 사용한다.
5. 배차담당자는 기존 TMS 또는 CSV·대시보드에서 예상 지연시간과 위험등급을 확인한다.
6. 데이터가 부족하거나 모델 불확실성이 크면 추천 대신 ‘판단 보류’와 근거를 표시한다.

### ‘출차시간’ 표현의 최종 처리

출차시각은 분류·상차 등 다른 운영조건에 묶여 있으므로 핵심 산출물에서 제외한다. 배차담당자에게 제공하는 정보는 다음으로 제한한다.

- 공사구간의 **권장 검토 시간대**
- 기존 배차계획에 더할 **예상 지연시간**
- 공사구간을 먼저 또는 나중에 처리할지 검토하는 **위험 신호**

최종 경로와 출발시각은 기존 TMS와 배차담당자가 결정한다.

## 3. 기존 시스템과의 역할 경계

### 확인된 기존 기능

공개된 공식 자료를 기준으로 다음 기능이 이미 존재한다.

- CJ대한통운 로이스 파슬: 예약·분류·배차·정산, 기사 앱, 기업고객 접수·조회 등 택배 프로세스 전반
- 카카오모빌리티 TMS: 대량 배송 경로 최적화, 배차 자동화, 실시간 도로상황과 미래 교통량을 반영한 배차계획
- 카카오 미래 운행정보 길찾기 API: 미래 출발시각 기준 경로, 도로공사 등 교통 장애의 통제정보 반영
- ITS 국가교통정보센터: 일부 본선·우회도로의 교통예측정보 API

이 비교는 공개 제품문서 범위에 한정한다. 모든 상용 TMS 기능을 전수조사했다는 뜻이 아니다.

### 대체하지 않는 기능과 담당할 기능

| 업무 | 기존 시스템 역할 | 본 제안 역할 |
|---|---|---|
| 주문·송장·상품·기사·차량·정산 관리 | 기존 택배시스템/TMS가 계속 담당 | 담당하지 않음 |
| 차량별 배차·경로 최적화 | 기존 TMS가 최종 결정 | 경로를 생성하지 않음 |
| 현재·미래 교통 기반 ETA | 상용 길찾기·TMS에도 존재 | 기준서비스와 기능 중복 여부를 검토 |
| 트램 공사계획과 물류노출의 결합 | 공개 제품문서만으로는 전용 기능 확인 불가 | 공사구간별 지연위험·배송노출을 함께 평가 |
| 공사 전·중·후 결과 축적 | 공사 시행기관의 업무와 TMS가 분리될 수 있음 | 계획–예보–사후평가를 같은 데이터 구조로 연결 |
| 외부 전달 | TMS마다 방식이 다름 | 공통 위험도 CSV·JSON 규격을 제안 |

### 실제로 대체할 수 있는 업무

이 서비스가 대체하려는 것은 TMS가 아니라 **담당자가 공사공지·교통지도·엑셀을 따로 확인해 경험으로 버퍼를 정하는 수작업**이다.

### 생산환경 연동에 대한 제한

현재 공개자료만으로 CJ대한통운·카카오모빌리티 등 특정 시스템이 외부 링크 페널티를 입력받는지 확인하지 못했다. 따라서 공모전 프로토타입은 다음 순서로 구현한다.

1. 독립형 대시보드
2. 계획경로 CSV 입력과 위험정보 CSV·JSON 출력
3. 범용 REST API 계약서
4. 실제 TMS 연동은 향후 업체 협의·보안·개인정보 검토 후 추진

특정 기업과 연동 또는 제휴가 확정된 것처럼 표현하지 않는다.

## 4. 트램 공사 이후의 지속·확장 구조

대전 트램 공식 추진현황은 2028년 철도종합시험 운행 및 개통을 향후계획으로 제시한다. 일정은 변경될 수 있으므로 제안서에는 확인일과 공식 출처를 함께 적는다.

| 단계 | 서비스 역할 | 주요 산출물 | 주장 상태 |
|---|---|---|---|
| 공사 전 | 계획된 통제구간·시간대의 물류위험을 사전 비교 | 시나리오별 예상 지연·노출 비교 | 통제상태가 충분히 학습될 때만 검증 가능 |
| 공사 중 | 실시간 교통으로 30분 후 위험을 예보 | 위험지도, 예상 지연, 우선관리 구간 | 이번 공모의 핵심 실증 |
| 개통 전환기 | 공사 종료 후 달라진 정상 교통패턴을 다시 설정 | 공사 중/개통 후 기준선 비교 | 2028년 이후 장기계획 |
| 개통 안정기 | 트램 전용 실시간 화면을 축소하고 사후평가 | 지속 병목·예측오차·정책평가 자료 | 향후 관측 필요 |
| 다음 도시변화 | 다른 계획공사·도로공간 재편에 데이터 구조 재사용 | 도시변화 물류 리스크 API | 운영주체·계획정보 연계 미확정 |

### 장기 확장의 범위

장기 확장은 다음처럼 **계획된 도시변화의 물류영향 평가**에 한정한다.

- 도로공사와 단계별 차로 통제
- 대중교통 인프라 공사와 개통 전환
- 대규모 행사·계획 통제로 인한 도로공간 변화
- 대규모 개발사업에 수반되는 계획 교통변화

사고·재난에 반응해 차량경로를 실시간 최적화하는 기능은 핵심범위에서 제외한다. 공고문에 제시된 2025년 수상작 ‘재난 및 돌발상황 대응을 위한 동적 차량경로 최적화’와의 중복을 피하기 위해서다.

### 정직한 종료조건

새로운 계획공사 데이터나 운영주체를 확보하지 못하면 트램 전용 서비스는 공사·개통 평가 후 종료한다. 이 경우에도 공개한 데이터 품질관리 방식, 공사–교통 링크 매핑표, 예측모델과 사후평가 방법은 다음 도시변화 분석에 재사용할 수 있다. 이를 ‘영구 서비스 도입’으로 과장하지 않는다.

## 5. 문제–개입–결과 구조

| 구분 | 확정 내용 |
|---|---|
| 대상 | 트램 공사의 교통대책을 검토하는 담당자 |
| 상황 | 공사계획, 실시간 교통, 상권정보가 서로 다른 자료와 단위로 존재함 |
| 문제 | 어느 공사구간·시간대가 도심배송 활동에 더 큰 위험을 만드는지 같은 기준으로 비교하기 어려움 |
| 원인 가설 | 공사통제 정보와 교통링크·물류노출 정보가 하나의 시공간 단위로 결합되지 않음 |
| 개입 | 공사 이벤트를 교통링크에 매핑하고 30분 후 통행시간과 배송노출을 함께 계산 |
| 작동 방식 | 예측 지연과 상권 노출을 분리 산출한 뒤, 근거·불확실성과 함께 위험정보로 제공 |
| 직접 산출 | 공사구간 위험지도, 시간대 비교표, TMS용 위험정보 파일/API 규격 |
| 기대 결과 | 담당자가 우선관리 구간과 시간대를 식별하고, 배차담당자가 기존 계획의 버퍼를 검토할 근거 확보 |
| 측정 | 예측 정확도, 고지연 탐지성능, 링크 매칭품질, 시나리오 구분력, 정보 전달 완결성 |

‘공사계획 담당자가 현재 물류영향을 고려하지 않는다’거나 ‘서비스 수요가 확인됐다’는 근거는 아직 없다. 이는 사용자 검증이 필요한 설계 가설이다.

## 6. 분석 질문과 검증 범위

### 이번 공모에서 실제로 검증할 질문

1. 지속성 기준모델보다 AI 모델이 트램 공사구간의 30분 후 링크 통행시간을 더 정확히 예측하는가?
2. 어떤 공사구간·시간대·도로 특성에서 높은 지연위험이 반복되는가?
3. 지연위험과 상가밀도 기반 배송노출이 동시에 큰 우선관리 구간은 어디인가?
4. 예측결과를 특정 TMS에 종속되지 않은 CSV·JSON 위험정보로 변환할 수 있는가?
5. 현재 상용 TMS·길찾기와 비교했을 때 본 제안이 추가로 제공하는 의사결정 정보는 무엇인가?

### 데이터 조건을 통과할 때만 검증할 질문

6. 같은 공사구간에서 통제시간대 또는 통제단계가 달라질 때 관측 통행시간 차이를 비교할 수 있는가?
7. 충분한 통제상태 변화가 있다면, 계획안 A/B의 상대적 위험을 모의실험할 수 있는가?

6~7번은 인과효과를 자동으로 뜻하지 않는다. 사전·사후기간, 비교구간, 다른 교통요인을 통제할 자료가 부족하면 ‘관측 비교’ 또는 ‘예상 시나리오’로만 표시한다.

### 현재 검증하지 않는 주장

- 트램 공사가 특정 배송 지연을 직접 유발했다는 인과관계
- 실제 택배 물량이나 주문량
- 실제 차량의 비용·연료·탄소·근로시간 절감
- 특정 TMS와의 생산환경 연동
- 공사계획 변경 또는 기관 도입 성과

## 8. 실제 사용 가능성 검증 게이트

### Gate 0 — 중복과 사용자 의사결정

**현재 판정: 통과.** `docs/competitor_matrix.md`에 공개 기능·수상작·역할경계를 기록했다. 사용자 수요 인터뷰는 미수행이므로 도입 수요는 가설로 유지한다.

완료 조건:

- 기존 TMS·미래 길찾기·공공 교통예측과 기능 비교표 작성
- 2024·2025 수상작 및 개발 중 7개 신서비스와 핵심기능 대조
- 본 제안의 출력이 ‘새 경로’가 아니라 ‘계획공사 물류위험 정보’임을 문서 전체에서 통일
- 교통대책 담당자가 바꿀 수 있는 입력과 결정항목을 한 화면으로 설명
- 사용자 수요를 확인하지 못하면 ‘사용자 검증 필요’ 상태를 유지

실패 시에는 30분 예측만을 차별점으로 주장하지 않는다.

### Gate 1 — 대전 교통 API 인증과 이용한도

**현재 판정: 차단됨.** 최신 데이터셋의 공식 한도는 개발계정 10,000건/일이나 저장된 자격증명으로 HTTP 403이다. 레거시 엔드포인트도 코드 30을 반환했다. 실패 자체를 검증 결과로 보존하고 공식 트램 페이지를 임시 관측원으로 사용한다.

완료 조건:

- 발급키를 넣은 호출에서 정상 응답코드와 실제 링크 레코드 수신
- 전체 링크 조회에 필요한 페이지 수와 호출량 실측
- 10분 간격 수집이 공식 일일한도 이내인지 계산
- 응답시각과 수집시각(KST), 링크 수, 중복·결측을 로그로 기록
- 10~20분 간격 복수 스냅샷에서 값이 실제로 갱신되는지 확인

10분 간격은 하루 144회다. 최신 명세의 10,000회 한도에는 여유가 있지만, 정상응답 전에는 실제 페이지 수·레코드 수·갱신성을 확정하지 않는다.

실패 시:

1. 키 인코딩·승인상태·요청 파라미터·HTTP/HTTPS 동작 확인
2. 제공기관 문의와 공식 대체경로 확인
3. 2026-07-16까지 정상 수집이 불가능하면 예측기능을 구현된 것처럼 쓰지 않고 정적 물류노출 분석으로 축소 여부 결정

### Gate 2 — 공사 이벤트와 교통링크 매핑

**현재 판정: 통과.** 매핑 18행의 참조·중복·결정값 검사는 오류 0건이다. 1공구의 `tram-z01-01`과 `tram-z01-02`는 공식 공사이미지 2건의 URL·PNG 규격·SHA-256과 공식 공지문 문구를 실시간 재검증해 공사범위를 high 신뢰로 확인했고, 각 이벤트에 medium 이상 관측구간을 포함했다. 다만 `tram-z01-01`의 같은 도로·지점·방향 라벨이 화면에 두 번 나타나 어느 발생순번이 실제 위치인지 구분할 좌표·공식 링크 ID가 없으므로 개별 관측구간 매핑은 medium으로 유지하고 두 항목을 보수적으로 함께 포함한다. Gate 통과는 **공사범위가 확인됐다는 뜻**이며 중복 라벨의 개별 위치까지 확정됐다는 뜻은 아니다.

매핑 우선순위:

1. 공식 페이지의 링크 식별자 직접 결합
2. 도로명·시작/종료 지점·방향 복합키
3. 표준 노드·링크 좌표를 확보하면 공간교차 보강
4. 모호한 링크는 수기검수 후 확정/보류/제외와 근거 기록

선택한 시범구간의 핵심 통제도로가 방향까지 식별되고, 모든 사용 링크에 매칭방법과 신뢰등급이 있어야 통과한다. 품질이 낮으면 신뢰도가 높은 2개 이상 공사구간으로 시범범위를 축소한다.

### Gate 2A — 배송노출 대리값

**현재 판정: 통과.** 공식 상가정보 2026-03-31 ZIP(341,021,001바이트)과 대전 CSV(78,607행)의 SHA-256을 고정했다. 상가 ID 중복, 타 시도, 잘못된 좌표는 모두 0건이다. WGS84 상가 좌표를 EPSG:5186으로 변환하고 공식 표준 링크·노드에서 250m 이내인 영업 중 상가를 계산해 `tram-z01-01=69`, `tram-z01-02=12`, `tram-z12-01=994`를 얻었다. 10개 시범 관측구간에 기준일·단위·공간/매핑 신뢰도와 함께 연결했다.

- 원본 ZIP·대전 CSV·표준 노드링크의 해시가 모두 기대값과 일치해야 실행
- 반경은 250m로 고정하고 거리계산 좌표계 EPSG:5186을 기록
- 공사 이벤트의 링크/노드 근거와 신뢰도를 별도 CSV로 보존
- 상가 수는 실제 택배 물량·주문량이 아닌 배송활동 노출 대리값으로만 표시
- 교통 위험과 임의 가중합하지 않고 별도 필드로 제공

### Gate 3 — 공사 상태정보의 분석 가능성

**현재 판정: 부분 통과.** 5개 이벤트를 구조화했으나 공사단계 A/B 학습에 필요한 상태변화 이력은 아직 없다. A/B 시뮬레이션은 장기기능으로 이동한다.

- 통제기간·시간·도로·방향·통제내용을 이벤트 기준표로 구조화
- 공사 단계가 실제로 바뀌는 시점과 교통관측기간이 겹치는지 확인
- 차로 축소 수 등 수치가 없으면 통제강도를 임의 생성하지 않음
- 변화상태가 한 종류뿐이면 A/B 공사계획 학습을 수행하지 않음

### Gate 4 — 원자료 품질

**현재 판정: 운영 통과.** 10분 수집, 원응답 gzip·SHA-256, SQLite 중복방지, 품질요약을 구현했다. 2026-07-15 01:41 점검 기준 12스냅숏·492행·41구간, 실패 0, 비정상/0 속도 0건이다.

- 일별 누락률·중복률·비정상 속도·통행시간·링크 수 급변 점검
- 연속 결측의 원인을 기록하고 임의 보간 전 제외기준 확정
- 원자료와 가공자료의 기간·행 수·변환 규칙·버전을 고정

### Gate 5 — AI 예측 유효성

**현재 판정: 평가 전(`insufficient_data`).** 코드와 합성 테스트는 통과했지만 실데이터 성능 주장은 금지한다. 2026-07-15 01:22 자동 최종화 점검은 10스냅숏·410행·1.337시간·1개 날짜·164예제로 네 최소조건 모두 미달이었다. 따라서 상태만 `pending`으로 기록했고 동결 DB·모델·최종 매니페스트는 만들지 않았다.

머신러닝이 지속성·동시간 기준모델을 이기지 못하면 ‘AI가 효과적’이라고 결론내리지 않는다. 구간별 성능과 불확실성을 함께 공개한다.

`scripts/finalize_submission.ps1`는 공식 데이터원·매핑 근거를 먼저 갱신하고, 모든 하드 게이트가 통과할 때만 SQLite 일관 복제본, 시간순 모델평가, 위험 CSV/JSON, 경로결합, 파일별 SHA-256 매니페스트를 만든다. AI가 기준모델을 이겨도 현재 관측 계약에 검증된 추론출력이 연결되기 전에는 `predicted_*`를 자동으로 채우지 않는다.

### Gate 6 — 시뮬레이터 주장 강도

**현재 판정: 축소 완료.** 제목을 관측·예보 프로토타입으로 낮췄고 현재 화면은 관측 전용이라고 명시한다.

- 시간대 비교는 실제 관측·예측값으로 수행
- 공사단계 A/B 비교는 서로 다른 통제상태 자료가 충분할 때만 수행
- 비교조건이 없으면 ‘영향 시뮬레이터 구현 완료’가 아니라 ‘위험 예보 프로토타입 + 확장 설계’로 표현하고 최종 제안명도 같은 수준으로 수정
- 관측 비교를 공사 인과효과로 바꾸지 않음

### Gate 7 — TMS 독립 연동

**현재 판정: 프로토타입 통과.** 샘플 경로 3행이 모두 현재 위험정보와 결합되어 CSV·JSON으로 출력됐다. 생산 TMS 연동은 미검증이다.

- 샘플 계획경로 CSV를 입력받아 위험정보 CSV·JSON 생성
- 필수 필드·단위·결측·오류응답 규격 작성
- 특정 업체 API 없이도 독립형 데모가 동작
- 생산 TMS 연동은 미검증 상태로 표시

## 9. 수집·가공·서비스 파이프라인

~~~text
공사 이벤트 기준표 ───────────┐
대전 실시간 교통 10분 자료 ───┼─ 링크×시간 패널 ─ 30분 통행시간 예측 ─┐
기상·도로·노드/링크 ─────────┘                                      │
상가정보 ─ 구간별 배송노출 대리값 ───────────────────────────────────┼─ 구간·시간 위험정보
                                                                      │
계획 시나리오 ─ 시간대/통제상태 비교 ────────────────────────────────┘
                                                                      │
                                      대시보드 / CSV·JSON / 향후 TMS API
~~~

### 원자료 보존

- 교통 원응답은 수집일·시각별로 수정하지 않고 보존한다.
- API 기준시각과 수집기가 부여한 collected_at_kst를 분리한다.
- 키와 전체 요청 URL은 로그에서 마스킹한다.
- 공사 이벤트 기준표에는 event_id, 공구, 도로명, 시작·종료 지점, 방향, 통제내용, 기간, 출처 URL, 확인일을 넣는다.
- 수기 분류와 모델 산출물을 같은 열에 덮어쓰지 않는다.

### 모델용 패널

- 기본 행 단위: linkID × 10분 시각
- 예측목표: 각 링크의 t+30분 통행시간(travelT)
- 시계열 특성: 현재·10·20·30분 전 속도·통행시간, 이동평균, 변동성
- 시간 특성: 시각, 요일, 주말, 출퇴근 시간대
- 도로 특성: 링크 길이, 방향, 도로명 또는 확인 가능한 도로유형
- 공사 특성: 공사구간 여부, 통제유형, 공구, 실제 확인된 공사상태
- 기상 특성: 강수·기온·풍속 등 실제 수신 변수
- 노출 특성: 공사링크 주변 상가 수 또는 밀도. 예측목표가 아니라 별도 노출지표로 유지

## 10. AI 모델·위험지표·검증 방법

### 비교 모델

1. **지속성 기준모델**: 현재 통행시간을 30분 후 값으로 사용
2. **동시간 기준모델**: 학습기간의 같은 요일·시간대 대표값 사용
3. **AI 후보**: scikit-learn 히스토그램 그래디언트부스팅 회귀

딥러닝은 짧은 수집기간에서 우위가 확인되지 않았으므로 기본안에서 제외한다. 재현성과 설명가능성을 우선한다.

### 데이터 분할

- 임의 행 분할을 금지하고 날짜 순서대로 학습–검증–시험기간을 분리
- 최종 시험기간은 모델·임계값 선택에 사용하지 않음
- 수집기간이 짧다는 한계를 공개하고 계절·연중 일반화 금지
- 가능하면 날짜 단위 순차검증으로 결과 변동성 확인

### 예측 평가

- 주지표: MAE
- 보조지표: RMSE 또는 WAPE
- 고지연 위험 탐지: Precision, Recall, F1
- 공사구간·비공사구간, 시간대별 성능 분리
- 공사·기상 변수를 뺀 제거실험
- SHAP 또는 변수중요도는 예측근거로만 해석하고 인과효과로 해석 금지

### 위험정보 구성

위험정보는 서로 다른 성격의 값을 합치기 전에 각각 제시한다.

- **예측 지연**: 예측 통행시간과 링크별 정상 통행시간의 차이
- **배송노출 대리값**: 링크 주변 상가 수·밀도
- **통합 위험등급**: 지연과 노출을 결합한 우선순위

정상 통행시간 정의, 두 값의 정규화 방식과 결합가중치는 탐색분석 후 고정하고 민감도 분석을 남긴다. 임의 가중치를 숨기지 않는다.

### 시나리오 비교

이번 공모에서 우선 검증할 시나리오는 ‘같은 공사상태에서 통제·통과 시간대를 달리했을 때의 예상 위험’이다. 통제단계별 관측이 충분하면 공사단계 비교를 추가한다. 충분하지 않으면 계획안 A/B 결과를 가상 성과처럼 제시하지 않는다.

### 효과 측정의 한계

모의실험에서 계산한 차이는 모델과 관측범위 안의 **추정치**다. 실제 차량 운행기록이 없으므로 배송시간 절감실적으로 표현하지 않는다.

## 11. TMS 독립 위험정보 규격

### 프로토타입 입력

- scenario_id
- event_id
- link_id 또는 좌표·도로명
- valid_from / valid_to
- 통제내용
- 선택 입력: 계획 통과시각 또는 계획경로 CSV

### 프로토타입 출력

- link_id
- forecast_at / target_at
- predicted_travel_time_sec
- baseline_travel_time_sec
- expected_delay_sec
- risk_grade
- exposure_proxy
- cause_features
- confidence_or_warning
- source_updated_at

필드명과 단위는 구현 후 실제 응답에 맞춰 확정한다. confidence는 통계적으로 정의할 수 있을 때만 제공하고, 그렇지 않으면 데이터 품질 경고로 대체한다.

### 화면·파일 산출물

1. **공사계획 화면**: 공구·통제기간·도로·방향과 데이터 상태
2. **30분 위험지도**: 예측 지연, 노출 대리값, 갱신시각
3. **시간대 비교표**: 후보 시간대별 위험과 판단근거
4. **TMS 전달 샘플**: 계획경로 CSV 입력, 위험정보 CSV·JSON 출력
5. **사후평가표**: 예측과 실제의 오차, 공사단계별 관측 비교

### 제품범위에서 제외할 것

- 주문·송장·기사·차량·정산·배송완료 관리
- 차량별 완전자동 경로최적화
- 재난·돌발상황 동적 차량경로 재계산
- 시민용 내비게이션
- 특정 기업 시스템과 연동됐다는 미확인 주장
- 실측하지 않은 비용·연료·탄소·배송시간 절감률
- 생성형 AI가 교통 예측값을 만든다는 설명

## 12. 생성형 AI 활용 기록

두 종류의 AI를 구분한다.

- **서비스 작동 AI**: 링크별 30분 후 통행시간을 예측하는 CatBoost/LightGBM
- **분석 보조 AI**: 아이디어 구체화, API 검증, 코드 초안·리뷰, 경쟁기능 조사, 결과 해석의 반론 점검, 문서 초안에 사용하는 생성형 AI

공고문이 요구하는 AI 도구명, 프롬프트 3개 이상, 결과물 검토·수정, 참가자 기여를 위해 docs/ai_usage_log.md에 다음을 기록한다.

- 사용일시, 단계, 실제 도구명과 화면 표시 모델/버전
- 실제 프롬프트 원문과 입력자료 범위
- AI 출력 요약
- 사람이 확인한 공식 근거
- 채택·수정·기각 내용과 이유
- 수정 전후 결과, 참가자가 직접 수행한 작업

### 이미 확보한 비판적 수정 사례

1. AI 초안의 ‘권장 출차시간’을 사용자가 문제 삼아, 출차가 아닌 예상 지연·위험 신호로 수정했다.
2. 사용자가 공사 종료 후 지속성을 문제 삼아, 트램 전용 배차도구에서 전주기 물류영향 모듈로 재설계했다.
3. 공식 제품문서를 확인하니 상용 TMS에도 미래 교통·배차기능이 있어 ‘TMS 대체’ 주장을 폐기했다.
4. 과거 교통상품의 페이지 수정일을 최신 관측기간으로 오해하지 않도록 핵심 학습자료에서 제외했다.
5. 2025년 동적 경로 최적화 수상작과 겹치지 않도록 재난·돌발 경로최적화를 제품범위에서 제외했다.

생성형 AI 답변은 근거로 사용하지 않는다. 공식 문서, 실제 API 응답, 원자료와 코드 실행결과로 재검증한다.

## 17. 완료 정의

다음 조건을 모두 충족해야 이번 공모 범위의 검증을 완료한 것으로 본다.

- 대전 교통 API를 실제 호출하고 성공 또는 실패·대체경로를 재현 가능하게 기록함
- 원자료 기간·행 수·누락률과 공사 이벤트–링크 매칭품질을 공개할 수 있음
- 최소자료가 충족되면 30분 예측모델을 시간순 시험자료에서 기준모델과 비교하고, 미충족이면 예측 주장을 잠금
- 지연위험과 배송노출 대리값을 분리해 산출함
- 샘플 계획경로를 위험정보 CSV·JSON으로 변환함
- 특정 TMS를 대체하거나 실제 연동했다고 표현하지 않음
- 공사 A/B 비교의 데이터 조건과 주장 한계를 명시함
- 공사 중–개통 전환–다음 계획공사로 이어지는 재사용·종료조건을 설명함
- 2024·2025 수상작과 7개 신서비스의 중복 검토표가 있음
- 모든 수치가 원자료·코드·표로 역추적됨
- 동결 SQLite와 모델·결과·근거 파일의 크기·SHA-256 매니페스트가 생성됨
- 대리변수·추정·계획·확인사실을 구분함
- 공고문·양식의 제출조건과 AI 활용기록을 최종감사함
