# D안 2차 승인용 개요

- 상태: 본문 작성 전 사용자 승인 대기
- 공모분야: 지정3 도시 변화 대응
- 선택안: 트램 공사공지를 프로그램이 읽을 수 있는 데이터로 바꾸기

## 권장 제목

**트램 공사공지를 배송 프로그램용 표로 바꾸기**

부제: AI와 국가표준 도로번호를 이용한 대전 트램 공사정보 변환

## 첫 문단

대전시는 트램 공사 장소와 차로 통제 내용을 홈페이지에 글과 그림으로 알리고 있습니다. 택배회사가 예정된 운행이 공사구간을 지나는지 미리 확인하려면, 이 공사정보를 배송 프로그램 안의 도로와 연결해야 합니다. 그러나 확인한 공개 화면에는 그 연결에 필요한 국가표준 도로번호가 없습니다. 이 제안은 AI가 공사 내용을 표로 정리하고, 사람이 확인한 도로번호를 붙여 배송 프로그램이 가져갈 수 있는 데이터로 만드는 방법입니다.

## 이 제안이 하는 일과 하지 않는 일

### 하는 일

- 홈페이지의 공사장소·기간·방향·통제차로를 같은 형식의 표로 정리
- 공사장소와 국가표준 도로번호의 연결 후보 제시
- 후보를 `확인·재확인·제외`로 나눔
- 원문 URL·근거 문구·마지막 확인시각을 함께 저장
- 확인된 데이터만 CSV·JSON·GeoJSON으로 내보내는 형식 제안

### 하지 않는 일

- 배송경로 생성
- 우회도로 선택
- 배송 지연 예측
- 현재 차량 속도를 이용한 공사 효과 판단
- 주변 상가 수를 배송량으로 해석
- 실제 대전시·택배회사 시스템 연동을 완료했다고 주장

## 제안서 개요

### 1. 주제선정

- 대전 트램 공사정보는 공사위치·기간·차로통제를 글과 이미지로 제공한다.
- 확인한 공개 화면에는 공사위치와 국가표준 도로번호를 함께 제공하는 다운로드 파일이나 공개 API가 표시되지 않는다.
- 배송 프로그램이 공사정보를 재사용하려면 사람이 장소를 다시 찾아 도로 데이터와 연결해야 한다.
- 목적은 새 배송서비스를 만드는 것이 아니라, 공사공지를 기존 지도·배송 프로그램이 가져갈 수 있는 입력자료로 바꾸는 것이다.
- 공개되지 않은 내부 데이터가 없다고 단정하지 않고, 공개 화면과 공식 검색 범위에서 확인하지 못했다고 쓴다.

### 2. 데이터 활용

- 대전시 트램 공사정보: 공사장소, 기간, 진행방향, 통제차로, 원문 이미지
- 국가교통정보센터 국가표준 노드·링크: 링크 ID, 노드 ID, 도로명, 방향, 길이
- 대전시 교통정보 OpenAPI: 구조화된 도로 데이터가 어떤 항목을 사용하는지 비교
- 실제 시범자료:
  - 공사정보 5건
  - 트램 교통화면의 도로구간 연결 후보 18개
  - 시범 포함 10개·재확인 6개·제외 2개
  - 국가표준 도로범위 연결 3건
  - 서대전역네거리~서대전네거리 양방향 표준 링크 8개
- 현재 속도 451건과 주변 상가 69·12·994곳은 배송 영향을 설명하지 못하므로 핵심 분석에서 제외한다.

### 3. AI 활용

- AI가 글·이미지에서 공사장소·기간·방향·통제차로의 후보값을 정리한다.
- AI가 장소명과 국가표준 도로자료를 비교해 도로 후보를 제시한다.
- AI는 후보를 고른 이유와 맞지 않는 부분을 원문 근거와 함께 설명한다.
- 날짜·필수항목·파일형식 검사는 정해진 규칙으로 처리한다.
- 최종 도로번호는 사람이 원문과 지도를 확인한 뒤 승인한다.
- AI 추출 정확도나 시간절감은 아직 시험하지 않았으므로 수치로 주장하지 않는다.

### 4. 분석방법과 결과

1. 공사공지 5건을 공사장소·기간·방향·통제차로 항목으로 정리했다.
2. 공사장소와 대전 트램 교통화면의 도로구간 이름을 비교해 후보 18개를 만들었다.
3. 도로명·장소·방향·공사기간의 근거를 대조했다.
4. 10개는 시범 분석에 포함하고, 6개는 근거 부족으로 재확인, 2개는 공사종료 여부와 장소 불일치로 제외했다.
5. 국가표준 노드·링크를 사용해 공사범위 3건에 표준 도로 참조를 붙였다.
6. 서대전역네거리~서대전네거리는 계백로 양방향 8개 표준 링크로 재현했다.
7. 이 결과를 바탕으로 표의 필수항목과 `확인·재확인·제외` 절차를 제안한다.
8. 18개 후보와 국가표준 도로 연결 3건은 서로 다른 분석이므로 하나의 성과처럼 합치지 않는다.

### 5. 타당성 및 차별성

- Waze·TomTom·카카오도 공사·도로통제 정보를 구조화하거나 경로에 반영하므로 아이디어 자체가 최초라고 주장하지 않는다.
- 차별성은 대전 트램 공개공지와 국가표준 노드·링크를 실제로 연결해 본 지역 시범사례에 둔다.
- AI가 가장 비슷한 도로를 자동 확정하지 않고, 원문 근거와 `재확인·제외` 상태를 남기는 방식을 강조한다.
- 실제 사용자 인터뷰와 시스템 연동이 없으므로 현장 효용은 후속 검증으로 분리한다.

### 6. 발전가능성

- 1단계: 현재 5건의 정답표를 사람이 검토하고 AI 항목 추출 정확도 측정
- 2단계: 14개 공구 전체 공사정보로 확대
- 3단계: 대전시 담당부서에 별도 구조화 데이터 존재 여부와 공개 가능 항목 확인
- 4단계: 지도·물류 프로그램 개발자에게 필요한 필드와 파일형식 확인
- 5단계: 확인된 수요가 있을 때만 공공데이터 파일 또는 API 시범 제공
- 중단 기준: 기존 공식 API가 같은 항목을 이미 제공하거나, 표준 도로 연결 정확도가 기준에 미달하면 제안을 축소하거나 종료

## 제안서에 넣을 시각자료

1. `현재 공지 → 제안 표` 비교
   - 왼쪽: 공사 글·이미지
   - 오른쪽: 장소, 기간, 방향, 통제차로, 표준 도로번호, 확인상태
2. `AI 후보 제시 → 사람 확인 → 파일 저장` 흐름
3. 실제 검증 결과
   - 공사 5건
   - 도로이름 후보 18개: 포함 10·재확인 6·제외 2
   - 표준 도로범위 3건

## 분석 과정 보고서 개요

### 1. 데이터 수집

- 공식 트램 페이지·공사이미지·국가표준 노드·링크의 수집일·출처·해시
- 공사정보 5건과 도로 후보 18개의 생성·검토 과정
- 현재 속도와 상가자료를 검토했지만 배송 영향을 증명하지 못해 제외한 이유

### 2. 활용 AI 도구

- Codex: 자료 조사, 항목 정리, 도로 후보 검토, 결과파일 확인, 아이디어 진단, 문서 초안
- AI가 한 작업과 사용자가 한 판단을 구분

### 3. 주요 프롬프트

- 공사공지 항목 정리
- 공사장소와 도로 후보 연결
- 근거 부족 후보 분리
- 기존 아이디어의 논리와 쉬운 문장 검토

### 4. AI 결과 검토·수정 사례

- `근거제약형 AI`라는 이해하기 어려운 기술명 삭제
- C안의 `택배 운행 조사`와 현재 속도 비교가 목적을 설명하지 못한다는 사용자 지적으로 C안 폐기
- 18개 후보를 모두 국가표준 도로번호로 연결한 것처럼 읽힐 수 있는 표현 수정

### 5. 팀 기여

- 사용자:
  - 아이디어 방향 선택
  - 이해하기 어려운 첫 문장과 용어 지적
  - 실제 현장 사용 가정의 불확실성 지적
  - C안의 목적과 현재 속도 지표에 문제 제기
  - D안 최종 선택
  - 제출 전 핵심 수치·출처·해석 검토 예정
- AI:
  - 자료 조사·데이터 정리·분석·진단·초안 작성
- 사용자 검토가 끝난 항목만 최종 기여 내용에 확정 기록

## 승인 뒤 작성할 문서

- 사실 그대로의 분석 과정 보고서
- 핵심 수치·출처 사용자 확인표
- 기존 원고에서 삭제할 주장 목록
