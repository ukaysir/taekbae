## 1. 주제선정 [2쪽]

### 1.1 해결하려는 문제

트램 공사정보는 공지문과 이미지, 교통상태는 별도 화면에 흩어져 있다. 교통대책 담당자가 배송 영향을 검토하려면 공사 위치와 방향을 도로구간에 다시 맞추고 주변 교통과 배송노출을 따로 확인해야 한다. 배차담당자도 지역 공사정보를 기존 TMS 경로와 직접 연결하기 어렵다.

이 제안은 비정형 공사정보를 도로구간별 배송위험 정보로 바꾼다. 근거제약형 AI는 공사 위치·기간·방향·차로통제를 구조화한 뒤 표준 도로링크·교통상태·상가노출 근거를 붙여 위험 브리핑을 작성한다. 근거제약형 AI는 제공된 공식자료와 허용된 필드 안에서만 답하며, 근거가 부족하면 빈칸을 추정하는 대신 판단을 보류한다.

### 1.2 사용자와 결과물

교통대책·공정 담당자가 먼저 AI가 정리한 공사 이벤트와 근거를 확인하고 구간 매핑을 승인한다. 시스템은 승인된 정보만 현재 교통상태·배송노출과 결합해 택배 영업소와 도시배송 배차담당자에게 다음 결과를 제공한다.

- 공사구간·방향·기간·부분차로 통제정보
- 현재 교통상태와 데이터 갱신시각
- 250m 영업 중 상가 수와 공간·매핑 신뢰도
- 원문 URL과 근거 ID가 붙은 위험 브리핑
- 기존 TMS가 읽을 수 있는 CSV·JSON 위험정보

주문·송장·기사·차량·정산과 최종 경로는 기존 TMS가 담당한다. 본 모듈은 출차시각이나 우회경로를 결정하지 않는다.

## 2. 데이터 활용 [3쪽]

### 2.1 교통관측과 품질

1·12공구의 고정 관측자료는 2026-07-18 12:50~16:24의 11개 스냅숏, 451행, 41구간이다. 수집 성공 11건과 수정 전 TLS 실패 1건을 기록했다. 비정상·0속도는 0건, 중앙 수집간격은 10.525분이다. 3.576시간의 자료이므로 AI 예측 성능이 아니라 데이터 구조와 품질관리 절차를 확인하는 데 쓴다.

### 2.2 공사 이벤트와 도로구간

공사 이벤트 5개와 후보 교통구간을 연결해 18행의 매핑 기준표를 만들었다. 시범 포함 10개, 후보 6개, 제외 2개다. 공식 이미지 2건과 공지문 1건으로 동부여성가족원~영진로얄아파트와 읍내삼거리의 공사범위를 확인했다. 중복 라벨은 공구번호와 화면 발생순번으로 분리하고 신뢰도를 낮췄다.

서대전역네거리~서대전네거리는 표준 링크상 순방향 4개 654.016m, 역방향 4개 650.315m이며 도로명은 모두 계백로였다. 테미고개는 정확한 관측라벨을 찾지 못해 시범구간에서 제외했다.

### 2.3 배송노출 대리값

대전 상가정보 78,607행은 좌표가 모두 유효했고 중복 상가 ID·타 시도·잘못된 좌표는 0건이었다. WGS84 좌표를 EPSG:5186으로 바꾼 뒤 공사 점·링크에서 250m 이내인 영업 중 상가를 계산했다. 계족로 69곳, 읍내삼거리 12곳, 계백로 994곳을 10개 시범구간에 연결했다.

상가 수는 배송활동 노출 대리값이며 실제 택배 물량이 아니다. 교통 위험과 합산하지 않고 기준일·단위·신뢰도를 별도 제공한다.

## 3. AI 활용 [4쪽]

### 3.1 핵심 AI 기능

핵심 AI는 공사정보를 읽어 근거가 있는 배송위험 브리핑으로 바꾼다.

1. 공지문·공사이미지에서 위치, 도로명, 기간, 방향, 통제차로를 추출한다.
2. 표준 링크와 관측라벨 후보를 받아 일치 근거와 충돌을 설명한다.
3. 교통상태·상가노출·신뢰도를 입력받아 담당자용 위험카드를 작성한다.
4. 모든 출력에 원문 URL과 근거 ID를 붙인다.
5. 필수 근거가 없거나 라벨이 중복되면 `판단 보류`로 돌려보낸다.

숫자 계산과 공사구간 확정은 AI에 맡기지 않는다. 거리계산·좌표변환·집계·스키마 검사는 규칙에 따라 처리하고, 담당자가 AI의 추출·매핑 결과를 최종 승인한다.

### 3.2 실제 분석에 활용한 AI

OpenAI Codex는 아이디어 비교, 공식자료 조사 보조, 데이터 처리 절차 설계, 반론 점검과 문서 초안에 활용했다. 생성형 AI의 답변은 출처로 인용하지 않고 공식문서·실제 HTTP 응답·파일 해시·고정 분석결과로 대조했다.

프롬프트는 주제 선택, API 실제 사용성, 출차시간의 의사결정 주체, TMS 대체 여부, 시간누수 없는 예측평가, 서비스 안의 AI 역할을 검토하는 데 사용했다. 참가자는 AI가 제안한 출차시간과 독립 배차도구를 삭제했다. 데이터가 부족한 30분 예측은 장기기능으로 옮기고 근거제약형 AI를 핵심 기능으로 확정했다.

## 4. 분석방법 [5~6쪽]

### 4.1 작동 구조

`공사공지·이미지 → AI 필드 추출 → 규칙 기반 후보 도로구간 생성 → AI 근거 비교·판단 보류 → 담당자 승인 → 교통·상가 데이터 결합 → 근거가 붙은 위험 브리핑·CSV`

AI는 비정형 문장을 구조화하고 근거를 설명한다. 좌표·거리·행 수·해시는 정해진 계산 절차로 구하며, 최종 구간은 담당자가 선택한다. 역할을 이렇게 나누면 생성형 AI가 숫자를 임의로 만들거나 잘못된 공간매핑을 확정하는 일을 막을 수 있다.

### 4.2 AI 평가계획

현재 보유한 수기 기준표를 소규모 정답셋으로 사용한다. 공사 이벤트 5개는 필드 추출 평가에, 매핑 18행은 후보 판정 평가에 쓴다. 측정지표는 다음과 같다.

- 도로명·기간·방향·통제내용의 필드별 정확 일치율
- 모든 위험문장에 근거 ID가 붙은 비율
- 입력 근거에 없는 필드나 숫자를 만든 비율
- 중복·충돌 사례에서 판단을 보류한 비율
- 담당자가 수정한 필드 수와 이벤트당 검토시간

이 지표로 AI가 공사정보 해석을 얼마나 정확하고 검토 가능하게 보조하는지 확인한다. 실행하지 않은 평가값은 성과로 제시하지 않는다.

### 4.3 향후 30분 예측

30분 교통예측은 장기 확장기능이다. 최소 288스냅숏·48시간·3개 날짜·5,000예제를 확보한 뒤 시간순으로 학습·시험을 나눈다. 지속성·동시간 기준모델과 히스토그램 그래디언트부스팅을 같은 시험구간에서 MAE로 비교하며, AI가 기준모델을 이기지 못하면 예보를 활성화하지 않는다. 현재 판정은 `insufficient_data`다.

## 5. 타당성 및 차별성 [7~8쪽]

### 5.1 기존 서비스와의 차이

| 기존/선행 | 주요 기능 | 본 제안의 역할 |
|---|---|---|
| 카카오 TMS | 자동배차, 경로최적화, 미래교통, ETA | 경로를 만들지 않고 부분차로 계획공사의 근거·노출 전달 |
| 카카오 미래 길찾기 | 미래 출발시각 경로, 전체차선 통제 반영 | 공사 전·중·후 사건과 근거 이력 관리 |
| CJ 로이스 파슬 | 예약·분류·배차·정산·기사앱 | 기존 코어시스템에 외부 위험정보 제공 |
| 2025 트램 심야 공동물류 | 트램 차량·거점을 물류에 활용 | 트램 공사가 기존 도로배송에 주는 위험을 분석 |
| 2025 돌발상황 동적경로 | 재난·돌발 경로최적화 | 계획공사 정보 구조화와 담당자 검토에 집중 |

이 제안의 차이는 비정형 계획공사 정보를 근거·신뢰도와 함께 물류 의사결정 정보로 바꾸는 데 있다. 근거가 약한 매핑은 AI가 보류하고 사람이 승인한다.

### 5.2 실행 가능성

이벤트 5개, 매핑 18행, 노출분석 3개 이벤트를 AI 평가 기준표로 쓴다. 주문·주소·기사 개인정보 없이 비식별 구간 ID만 다룬다. CSV 파일로 먼저 검증하고 이후 링크 ID 변환·인증·보안을 협의한다.

## 6. 발전가능성 [9~10쪽]

| 단계 | 핵심 작업 | 판단지표 |
|---|---|---|
| 1단계 | 5개 이벤트·18개 매핑으로 AI 추출·근거제시 시험 | 필드 일치율, 근거 연결률, 무근거 생성률, 보류율 |
| 2단계 | 교통대책·배차담당자 검토 | 수정 필드 수, 검토시간, 경고 이해도 |
| 3단계 | 실제 TMS 경로와 비식별 파일 교환 | 구간 매칭률, 스키마 오류율, 장애시 기본행동 |
| 4단계 | 장기 교통자료로 30분 예측 평가 | 기준모델 대비 MAE, 오경보·미탐지 |
| 5단계 | 다른 계획공사·행사·도로개편으로 확장 | 사건 스키마 재사용률, 신규 지역 매핑 성공률 |

공사페이지가 `event_id`, 표준 `link_id`, 방향, 통제차로 수, 계획·실제 시각을 제공하면 지도와 TMS가 같은 사건을 참조할 수 있다. AI는 누락값을 추정하지 않고 판단을 보류한다.
