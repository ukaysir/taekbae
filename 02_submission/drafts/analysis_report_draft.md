실제 활용 데이터는 대전 트램 공식 공사·교통 페이지, ITS 표준 노드·링크, 소상공인시장진흥공단 상가정보다. 대전교통정보 OpenAPI와 기상청 ASOS는 실제 호출이 HTTP 403이어서 분석 입력에서 제외했다.

## 1. 데이터 수집·전처리·분석

대전 트램 1·12공구의 구간별 교통상태와 속도를 관측했다. 작업 당시 원문 HTML은 수집시각·URL·SHA-256과 함께 gzip으로 보존했고, 정규화 관측치는 SQLite에 저장했다. 같은 라벨은 임의로 합치지 않고 공구번호와 화면 발생순번으로 나눴다. 2026-07-18 12:50~16:24에 고정한 자료는 11스냅숏·451행·41구간이다. 수집 성공 11건, 수정 전 TLS 실패 1건, 비정상·0속도 0건이며 중앙 수집간격은 10.525분이었다. 추가 수집은 끝냈고 현재 정리본에는 원응답과 SQLite를 넣지 않았다.

공사 이벤트 5개는 위치·기간·통제내용·방향·출처를 수기 기준표로 정리했다. 공식 공사이미지 2건은 URL·PNG 규격·SHA-256을, 공지문 1건은 기대문구를 확인했다. 공사 이벤트와 후보 교통구간의 매핑은 18행이다. 시범 포함 10개, 후보 6개, 제외 2개이며 각 행에 신뢰도와 판단이유를 남겼다. 서대전역네거리↔서대전네거리는 표준 링크상 순방향 4개 654.016m, 역방향 4개 650.315m였고 도로명은 모두 계백로였다.

2026-03-31 기준 대전 상가정보 78,607행은 좌표가 모두 유효했고 중복 상가 ID·타 시도·잘못된 좌표는 0건이었다. WGS84를 EPSG:5186으로 변환해 공사 점·링크에서 250m 안의 영업 중 상가를 계산했다. 계족로 69곳, 읍내삼거리 12곳, 계백로 994곳을 10개 시범구간에 연결했다. 상가 수는 실제 물량이 아니라 배송활동 노출 대리값으로 구분했다.

참고자료와 해시는 `03_analysis/references/sources.md`에서 관리한다.

## 2. 활용 AI 도구

**OpenAI Codex(GPT-5 기반 개발 에이전트)**는 아이디어 비교, 공식자료 조사 보조, 데이터 처리 절차 설계, 반론 점검, 결과 해석과 문서 초안 작성에 활용했다. 생성형 AI 답변은 근거로 인용하지 않았다. 공식문서·실제 HTTP 응답·파일 해시·고정 분석결과와 대조한 뒤 참가자가 채택 여부를 결정했다.

서비스 안에는 **근거제약형 생성 AI**를 적용한다. 공지문과 공사이미지에서 위치·기간·방향·통제차로를 뽑고, 도로링크 후보가 맞는 이유를 설명한 뒤 출처가 붙은 위험 브리핑을 작성한다. 입력 근거가 없거나 라벨이 충돌하면 값을 만들지 않고 판단을 보류한다. 이 기능은 제안 설계이며 성능평가를 마친 기능으로 쓰지 않는다.

## 3. 주요 프롬프트

### 프롬프트 1 — 주제와 차별성

- 활용단계: 공모주제 선정
- 입력: “지정공모 5개와 자유공모를 적합성·데이터 접근성·차별성으로 비교하고, 공식 붙임과 선행수상작을 근거로 아이디어를 제안하라.”
- 결과·검토: 트램 공사로 인한 도심배송 위험을 선택했다. 트램 차량 활용과 자체 경로최적화는 기존 수상작과 겹쳐 제외했다.

### 프롬프트 2 — 데이터 실제 사용성

- 활용단계: 데이터 검증
- 입력: “API의 실제 호출·필드·한도·갱신주기와 대체경로를 검증하라. 실패를 사용 가능으로 쓰지 말라.”
- 결과·검토: 대전교통정보·KMA는 HTTP 403으로 제외하고, 실제 확보한 트램 페이지·표준 링크·상가정보만 분석에 사용했다.

### 프롬프트 3 — 공간매핑 반론

- 활용단계: 공사–교통구간 연결
- 입력: “이름이 비슷하다는 이유만으로 연결하지 말고 공식 이미지·공지·표준 링크로 방향과 범위를 검증하라. 모호하면 제외하라.”
- 결과·검토: 테미고개 초기 추정을 폐기하고, 중복 라벨은 신뢰도를 낮추거나 보수적으로 포함했다.

### 프롬프트 4 — 시간누수 없는 예측평가

- 활용단계: 향후 모델 설계
- 입력: “t 시점까지의 값만 특성으로 만들고 t+30분을 목표로 하라. 시간순으로 기준모델과 비교하고 최소자료 미달이면 중단하라.”
- 결과·검토: 최소 288스냅숏·48시간·3일·5,000예제 게이트를 세웠다. 현재 자료는 `insufficient_data`로 판정하고 예측 성능을 결과에서 제외했다.

### 프롬프트 5 — AI 역할 재설계

- 활용단계: 핵심 기능 확정
- 입력: “이게 AI·데이터 활용 공모전인데 AI 기능이 들어가야 할 것 같다. 현재 데이터로 설명 가능한 AI 역할을 제안하라.”
- 결과·검토: 30분 예측 대신 공사정보 구조화·근거 연결·위험 브리핑을 핵심 AI로 제안했다. 참가자가 이 방향을 확정하고 30분 예측은 장기기능으로 옮겼다.

## 4. AI 결과물 검토·수정 과정

### 사례 1 — 권장 출차시간 삭제

AI는 위험이 낮은 시간대를 출차 추천으로 제시했다. 그러나 영업소의 분류완료, 상차, 기사·차량, 배송마감 조건이 없어 책임 있는 추천을 만들 수 없었다. 참가자는 출차시각을 삭제하고 위험정보만 기존 TMS와 배차담당자에게 전달하도록 범위를 바꿨다.

### 사례 2 — 예측 중심안 축소

초안은 30분 교통예측을 핵심 기능으로 뒀다. 고정자료가 11스냅숏·3.576시간·1일·41예제로 최소조건에 미달해 예측값과 성능 주장을 제외했다. 관측자료는 품질관리와 데이터 결합 검증에만 사용했다.

### 사례 3 — 서비스 AI를 현재 데이터에 맞게 재설계

참가자는 분석보조 AI만 강조하면 서비스 안에서 AI가 하는 일이 약해진다고 지적했다. 이에 비정형 공사정보를 구조화하고 근거가 붙은 위험 브리핑을 작성하는 AI로 중심을 옮겼다. 거리·좌표·집계는 정해진 절차로 계산하고, 근거가 부족한 판단은 AI가 보류하도록 역할을 나눴다.

## 5. 팀 기여 내용

- 참가자: 공모방향과 주제 선택, 출차시간·TMS 대체·AI 역할에 대한 문제 제기, 핵심 기능 확정, 최종 해석·표현·제출 판단
- AI 도구: 공식자료 탐색 보조, 데이터 검증 절차·모델평가 기준·근거제약형 AI 구조와 문서 초안 제안
- 공동 검증: API 응답코드, 공식이미지·상가파일 해시, 표준 링크와 공간거리, 수치·출처 일치 여부 확인
- 최종 원고·개인정보·동의·서명: `[대표자 및 팀원 입력·확인 필요]`

현재 분석은 공사정보 5개, 매핑 18행, 시범구간 10개를 AI 평가용 기준표로 준비했다. 다음 단계에서는 필드 정확 일치율, 근거 연결률, 무근거 생성률, 판단 보류율과 사람의 수정량을 측정한다. 예측 우월성·배송지연 인과효과·시간 및 비용 절감·실제 TMS 연동은 현재 분석결과로 주장하지 않는다.
