# UI·UX 요소와 표시 데이터 확정본

- 버전: v1.1
- 확정일: 2026-07-20 KST
- 범위: 확정 정보구조·화면요소·표시 데이터와 제출용 시각자료 제작 결과
- 디자인 반영: 제공된 레퍼런스의 밝은 회색 배경, 네이비 제목, 전기 파랑 강조, 민트 완료, 주황 확인필요, 둥근 카드·상태 배지·넓은 여백을 재해석

## 1. 제품 화면의 역할

화면은 공사자료를 사람이 검증하는 과정과 배송담당자가 확인하는 결과를 분리한다. AI가 공지문과 이미지를 먼저 정리하더라도 최종 도로구간은 담당자가 공식 원문과 지도를 확인한 뒤 선택한다. 검증이 끝난 정보만 배송위험 요약에 노출한다.

시범단계에서는 여러 값을 합친 단일 `AI 위험점수`를 만들지 않는다. 공사조건, 공식 화면의 현재 교통상태, 주변 상가 분포와 자료 갱신시각을 각각 보여준다. 이렇게 해야 배송담당자가 무엇 때문에 주의가 필요한지 직접 확인할 수 있다.

## 2. 사용자와 핵심 흐름

### 사용자 1 — 공사·교통정보 검토 담당자

- AI가 공지문·이미지에서 정리한 공사정보를 확인한다.
- 공식 원문과 지도에서 장소·방향·통제차로를 대조한다.
- 방향별 도로구간 후보마다 `이 구간 사용`, `추가 확인`, `제외` 중 하나를 선택한다.
- 검증이 끝난 공사정보를 배송위험 요약으로 공개한다.

### 사용자 2 — 택배 영업소·배차 담당자

- 운행 전에 공사구간과 통제기간을 확인한다.
- 현재 교통상태와 자료 갱신시각을 함께 확인한다.
- 주변 250m 영업 중 상가 수를 추가 확인 우선순위 자료로 본다.
- 필요한 경우 `근거 보기`에서 대전시 공지나 공사이미지를 확인한다.
- 최종 경로와 출차계획은 기존 운송관리시스템(TMS)에서 결정한다.

### 전체 흐름

`공사자료 게시 → AI 항목 정리 → 담당자 원문 확인 → 방향별 도로구간 선택 → 교통·상가정보 결합 → 배송위험 요약 공개 → 기존 TMS에서 최종 배차`

## 3. 필수 화면과 UI 구성요소

### 화면 A — 공사정보 검토 대기목록

목적은 새 공사자료와 재검토가 필요한 항목을 놓치지 않는 것이다.

| UI 요소 | 표시 내용 | 사용자 행동 |
|---|---|---|
| 상태 요약 | 검토 대기, 추가 확인, 검증 완료 건수 | 필요한 상태로 목록 필터링 |
| 검색·필터 | 공구, 도로·장소명, 공사기간, 검토상태 | 대상 공사 찾기 |
| 공사정보 목록 | 장소, 공사기간, 방향, 차로통제, 원자료 수, 마지막 확인시각 | 상세 검증 화면 열기 |
| 상태 배지 | `검토 대기`, `추가 확인`, `검증 완료`, `제외` | 처리 우선순위 확인 |
| 자료상태 안내 | 원문 접근 불가, 교통정보 오래됨, 필수항목 누락 | 정상자료와 예외 구분 |

목록의 기본 정렬은 `추가 확인 → 검토 대기 → 검증 완료 → 제외` 순서다. 색상만으로 상태를 구분하지 않고 상태명과 이유를 함께 표시한다.

### 화면 B — 공사정보·도로구간 검증

목적은 AI가 정리한 내용과 공식자료를 같은 화면에서 대조하고, 사람이 사용할 도로구간을 결정하는 것이다.

| 영역 | 필수 UI 요소 | 표시 데이터 |
|---|---|---|
| 공식자료 | 원문 미리보기, `원문 열기`, 자료번호, 출처기관, 확인시각 | 공지 URL, 공사이미지 URL, 자료 확인시각 |
| AI 정리결과 | 항목별 값과 연결 근거 | 장소, 도로명, 시작일, 종료일, 방향, 통제차로, 각 항목의 자료번호 |
| 도로구간 후보 | 후보 목록, 방향 표시, 비교이유, 근거상태 | 구간 ID, 구간명, 방향, 도로명·장소·공식지도와 일치한 이유 |
| 사람의 결정 | `이 구간 사용`, `추가 확인`, `제외` | 결정값, 결정이유, 확인자, 확인시각 |
| 예외 안내 | 모호한 이유를 설명하는 경고상자 | 같은 이름의 구간 반복, 방향 미기재, 원문 접근 실패, 좌표 불충분 |

`구간 매핑 승인`처럼 내부 작업용 표현은 화면에 쓰지 않는다. 담당자가 실제로 해야 할 행동을 그대로 버튼명으로 사용한다. 근거수준을 숫자 확률로 표시하지 않고 `근거 충분`, `추가 확인`, `근거 부족`과 이유를 함께 보여준다.

### 화면 C — 배송위험 요약

목적은 배차담당자가 운행 전에 공사와 주변 상황을 빠르게 확인하도록 하는 것이다.

| 영역 | 필수 UI 요소 | 표시 데이터 |
|---|---|---|
| 공사 요약 | 제목, 공구, 기간, 통제내용, 방향 | 공사장소, 시작·종료일, 통제차로, 진행방향 |
| 관련 도로구간 | 방향별 구간 목록 | 구간 ID, 구간명, 상·하행 방향 |
| 현재 교통 | 공식 상태 배지와 갱신시각 | `원활·지체·정체`, 관측속도, 원자료 갱신시각 |
| 주변 영업지점 | 수치, 거리기준, 기준일, 해석 안내 | 250m 안의 영업 중 상가 수, 상가정보 기준일, `실제 배송량 아님` |
| 근거 | 문장별 `근거 보기`, 자료번호 | 공지·이미지 URL, 확인시각, 사람이 검증한 상태 |
| 업무 연결 | CSV/JSON 내보내기 또는 기존 TMS로 전달 | 구간 ID, 비식별 경로 ID, 구간 순서, 계획 통과시각 |

첫 화면에서 가장 먼저 보일 내용은 `어디에서`, `언제까지`, `어느 방향과 차로가 통제되는가`이다. 현재 교통과 주변 상가 수는 그 아래에서 별도 근거로 보여준다. 주변 상가 수가 많다는 이유만으로 위험등급을 자동으로 높이지 않는다.

## 4. 화면 상태와 문구

| 상태 | 사용자에게 보일 문구 | 시스템 동작 |
|---|---|---|
| 검증 완료 | `공식자료와 도로구간 확인 완료` | 배송위험 요약에 공개 가능 |
| 추가 확인 | `도로 방향 또는 구간을 더 확인해야 합니다` | 배송위험 요약 공개 보류 |
| 근거 부족 | `공식자료만으로 관련 구간을 확인할 수 없습니다` | 값을 임의로 채우지 않고 제외 검토 |
| 교통자료 오래됨 | `현재 교통정보의 갱신시각을 확인하세요` | 마지막 값과 갱신시각을 함께 표시 |
| 교통자료 없음 | `현재 교통정보가 없습니다` | 속도·상태를 빈값으로 유지 |
| 예측 미제공 | `현재 관측정보이며 미래 예측이 아닙니다` | 예측시간·예상지연 필드를 표시하지 않음 |
| 주변 상가 값 없음 | `검증된 공간범위가 없어 주변 상가 수를 제공하지 않습니다` | 0으로 바꾸지 않고 빈값 유지 |
| 원문 접근 실패 | `공식 원문을 열 수 없어 확인이 필요합니다` | 새 검증 완료 처리 차단 |

## 5. 확정 데이터 목록

### 공사정보

| 표시명 | 내부 필드 | 출처 |
|---|---|---|
| 공사 ID | `event_id` | 내부 추적 ID |
| 공구 | `zone`, `contract_section` | 대전 트램 공사페이지 |
| 공사장소 | `location` | 대전 트램 공지·이미지 |
| 시작일·종료일 | `start_date`, `end_date` | 대전 트램 공지·이미지 |
| 통제내용 | `control_detail` | 대전 트램 공지·이미지 |
| 진행방향 | `direction` | 대전 트램 공지·이미지 |
| 자료상태 | `status_as_of_2026_07_14` | 공지에 표시된 계획과 확인기록 |
| 원문·확인시각 | `source_url`, `checked_at_kst` | 출처대장 |

### 도로구간 검증정보

| 표시명 | 내부 필드 | 설명 |
|---|---|---|
| 도로구간 ID | `segment_id` | 방향별 구간 식별자 |
| 도로구간명 | `segment_label` | 공식 교통화면 기반 이름 |
| 비교방법·이유 | `mapping_method`, `reason` | 도로명·장소·방향·공식지도와 비교한 방법과 이유 |
| 근거수준 | `confidence` | 화면에는 이유와 함께 쉬운 상태명으로 변환 |
| 사용결정 | `decision` | `include_pilot`, `candidate`, `excluded`를 사용자 문구로 변환 |
| 확인시각 | `checked_at_kst` | 사람이 근거를 확인한 시각 |

### 현재 교통정보

| 표시명 | 내부 필드 | 설명 |
|---|---|---|
| 공식 교통상태 | `observed_traffic_state` | 원활·지체·정체 |
| 관측속도 | `observed_speed_kmh` | km/h |
| 갱신시각 | `source_updated_at_kst` | 공식 화면의 관측 기준시각 |
| 정보성격 | `model_status`, `confidence_or_warning` | 현재는 관측이며 예측이 아님을 표시 |

### 주변 상가 분포

| 표시명 | 내부 필드 | 설명 |
|---|---|---|
| 주변 영업 중 상가 수 | `exposure_proxy` | 공사 지점·도로에서 250m 이내인 상가 수 |
| 거리기준 | `exposure_proxy_unit` | `active_store_count_within_250m` |
| 상가정보 기준일 | `exposure_source_date` | 현재 2026-03-31 |
| 공간근거 상태 | `exposure_confidence` | 검증된 공간범위가 없으면 값을 비워 둠 |

### 기존 TMS 전달용 최소정보

| 표시명 | 내부 필드 | 개인정보 여부 |
|---|---|---|
| 비식별 경로 ID | `route_id` | 개인정보 아님 |
| 경로 안의 순서 | `stop_order` | 개인정보 아님 |
| 도로구간 ID | `segment_id` | 개인정보 아님 |
| 계획 통과시각 | `planned_at_kst` | 개인정보 아님 |

주문번호, 송장번호, 주소, 수취인, 기사·차량 개인정보는 이 화면과 시범 연동 입력에 포함하지 않는다.

## 6. 디자인에 사용할 고정 사례 데이터

디자인 시안에는 아래 값만 실제 분석결과 예시로 사용할 수 있다. 현재 운영 중인 실시간 서비스나 성능검증 결과처럼 표현하지 않는다.

| 구분 | 확정값 | 기준·의미 |
|---|---|---|
| 공사정보 | 5건 | 대전 트램 1·12공구 정리본 |
| 도로 연결 후보 | 18개 | 시범 데이터 결합 10개, 담당자 재검증 6개, 연결 차단 2개 |
| 교통관측 | 11회·451개 기록·41개 구간 | 2026-07-18 12:50~16:24, 약 3시간 35분 |
| 상가정보 | 대전 78,607건 | 2026-03-31 기준 영업 중 상가 위치 분석 |
| 계족로 사례 | 69곳 | 공사범위에서 250m 이내 영업 중 상가 수 |
| 읍내삼거리 사례 | 12곳 | 공사지점에서 250m 이내 영업 중 상가 수 |
| 계백로 사례 | 994곳 | 공사범위에서 250m 이내 영업 중 상가 수 |
| 읍내삼거리 공사 | 2025-05-08~2026-11-30, 양방향 1~2개 차로 부분통제 | 공사계획 정보 |

특정 구간의 속도·교통상태를 시안에 넣을 때는 `03_analysis/results/current_risk.csv`의 관측시각과 값을 함께 표시한다. 그 값을 현재 시각의 실시간 상태처럼 재사용하지 않는다.

## 7. 넣지 않을 UI와 데이터

- 근거가 없는 종합 AI 위험점수와 정확도 백분율
- 검증되지 않은 30분 교통예측, 예상 지연시간과 도착시간
- 권장 출차시각, 자동 배차, 자동 경로변경 버튼
- 주변 상가 수를 배송건수·주문량·매출로 표현한 값
- 모호한 도로후보를 검증 완료처럼 보이게 하는 기본 선택
- `insufficient_data`, `include_pilot` 같은 내부 상태코드의 직접 노출
- 주문·송장·주소·수취인·기사·차량 개인정보
- 운영기관 협의, TMS 연동 완료, 실제 배송성과로 오해할 표현
