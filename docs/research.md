## 3. 확정 아이디어와 사용자

### 3.1 제안명과 핵심 정의

가제는 **대전 트램 전주기 AI 물류영향 시뮬레이터 — 트램을 첫 실증으로 한 도시변화 리스크 API**다. 다만 공사단계 A/B 비교를 뒷받침할 데이터가 부족하면 `시뮬레이터`라는 표현을 낮춰 **위험 예보 프로토타입 + 확장 설계**로 제출한다.

핵심 정의는 다음과 같다.

> 대전시·트램 공사 시행기관의 교통대책 담당자가 공사 통제계획과 실시간 교통을 결합해 시간대·구간별 물류통행 위험을 비교하고, 그 결과를 기존 TMS와 도심배송 배차담당자에게 위험도·예상 지연시간으로 제공하는 의사결정 지원 서비스다.

### 3.2 사용자와 역할

| 역할 | 대상 | 서비스가 돕는 판단 | 현재 상태 |
|---|---|---|---|
| 1차 의사결정자 | 대전시 또는 트램 공사 시행기관의 교통대책·공정계획 담당자 | 우선관리 구간, 통제 시간대, 사전안내 검토 | 제안 대상. 실제 수요·도입 협의 미검증 |
| 운영 수혜자 | 택배 영업소·도시배송업체 배차담당자 | 기존 배차에 더할 버퍼와 공사구간 통과순서 검토 | 사용성 검증 필요 |
| 연동 대상 | 기존 TMS·배차솔루션 | 링크·시간대별 위험정보 수신 | 공개 연동규격·제휴 가능성 미확인 |
| 간접 수혜자 | 배송기사 | 기존 도구 안에서 조정된 운행계획 확인 | 별도 앱을 요구하지 않는 방향 |

### 3.3 `출차시간`에 대한 결론

출차시각은 분류·상차·기사·차량 등 영업소 내부 조건의 영향을 받으므로 서비스가 직접 결정하지 않는다. 서비스 출력은 다음 세 가지로 제한한다.

- 공사구간의 권장 **검토 시간대**
- 기존 배차계획에 반영할 **예상 지연시간**
- 공사구간 처리순서를 재검토하게 하는 **위험 신호**

최종 경로와 출발시각은 기존 TMS와 배차담당자가 결정한다.

## 4. 트램 공사 이후의 활용 결론

트램 전용 화면을 영구 서비스로 주장하지 않는다. 공사 종료 후에는 개통 전환기 평가와 정상 교통패턴 재설정을 수행하고, 이후 다른 **계획된 도시변화**에 같은 데이터 구조와 분석 절차를 재사용한다.

| 단계 | 역할 | 상태 |
|---|---|---|
| 공사 전 | 계획된 통제구간·시간대의 위험 사전 비교 | 데이터 조건이 있을 때 검증 |
| 공사 중 | 실시간 교통으로 30분 후 링크 통행시간·위험 예보 | 이번 공모의 핵심 실증 |
| 개통 전환기 | 공사 중과 개통 후 기준선 비교, 정상패턴 재설정 | 장기 계획 |
| 개통 안정기 | 트램 전용 실시간 화면 축소, 예측오차·지속 병목 사후평가 | 향후 관측 필요 |
| 다음 도시변화 | 다른 도로공사·대중교통 공사·계획통제에 구조 재사용 | 운영주체와 계획정보 연계 미확정 |

새로운 계획공사 데이터나 운영주체를 확보하지 못하면 트램 전용 서비스는 공사·개통 평가 후 종료한다. 이 종료조건을 숨기지 않고, 재사용 가능한 산출물을 공사–교통 링크 매핑표, 데이터 품질관리 방식, 예측모델, 사후평가 방법으로 한정한다.

`최초`, `유일`, `완전 대체`는 비교 근거가 부족하므로 사용하지 않는다.

### 6.1 핵심 API 기록

- 대전 교통 엔드포인트: `http://openapitraffic.daejeon.go.kr/traffic/rest/getTrafficInfoAll.do`
- 예상 주요 필드: `linkID`, `roadName`, `speed`, `travelT`, `congestion`, 시작·종료 노드, 링크 길이
- 공공데이터포털 키 환경변수: `DATA_GO_KR_SERVICE_KEY`
- 도시교통정보센터 키 환경변수: `UTIC_API_KEY`
- API 키는 문서, 코드, Git, 화면 캡처, 전체 요청 URL에 남기지 않는다.

현재까지 **인증키를 사용한 정상 호출, 실제 링크 레코드 수신, 호출량·갱신주기 실측은 완료되지 않았다.** 따라서 API가 생산적으로 사용 가능하다고 아직 확정할 수 없다.

## 7. 분석·AI 설계

### 7.1 데이터 단위와 예측 목표

- 기본 행: `linkID × 10분 시각`
- 예측 목표: 각 링크의 `t+30분 통행시간(travelT)`
- 입력 후보: 현재·과거 속도와 통행시간, 이동평균·변동성, 시간·요일, 링크 특성, 실제 확인된 공사상태, 기상
- 상가정보: 예측 입력의 핵심 물동량이 아니라 별도의 **배송노출 대리값**

### 7.2 모델과 검증

1. 지속성 기준모델: 현재 통행시간을 30분 후 값으로 사용
2. 동시간 기준모델: 학습기간의 같은 요일·시간대 대표값 사용
3. AI 후보: CatBoost 또는 LightGBM 기반 비선형 회귀

임의 행 분할은 사용하지 않고 날짜 순으로 학습–검증–시험을 나눈다. 주지표는 MAE, 보조지표는 RMSE 또는 WAPE이며 고지연 탐지에는 Precision·Recall·F1을 사용한다. AI 모델이 기준모델을 이기지 못하면 AI가 효과적이라고 주장하지 않는다.

### 7.3 두 종류의 AI

- 서비스 작동 AI: 링크별 30분 후 통행시간을 예측하는 CatBoost/LightGBM 후보
- 분석 보조 AI: 아이디어 구체화, 코드 초안·리뷰, 경쟁기능 조사, 반론 점검, 문서 초안에 사용하는 생성형 AI

생성형 AI 답변 자체는 근거가 아니다. 공식 문서, 실제 API 응답, 원자료와 코드 실행결과로 재검증하고 실제 사용 프롬프트와 사람의 수정 판단을 별도 기록해야 한다.

## 8. 위험정보와 주장 한계

위험정보는 다음 세 값을 먼저 분리해서 보여준다.

- 예측 지연: 예측 통행시간과 링크별 정상 통행시간의 차이
- 배송노출 대리값: 링크 주변 상가 수 또는 밀도
- 통합 위험등급: 지연과 노출을 결합한 검토 우선순위

정상 통행시간 정의, 정규화 방식, 결합가중치는 탐색분석 이후 고정하고 민감도 분석을 남긴다. 다음 내용은 현재 검증하지 않았으며 사실처럼 쓰지 않는다.

- 트램 공사가 특정 배송 지연을 직접 유발했다는 인과관계
- 실제 택배 물량·주문량·차량 운행기록
- 실제 비용·연료·탄소·배송시간·근로시간 절감
- 특정 TMS와의 생산환경 연동 또는 기업 제휴
- 기관이 공사계획을 변경했거나 서비스를 도입했다는 성과
- 공사단계 A/B의 인과적 효과

## 10. 우선순위가 높은 조사 공백

1. **P0 / 제출 적합성:** 최신 정정공고, 참가 자격, 팀 구성, 모든 필수서명·첨부를 제출 직전 재확인한다.
2. **P1 / 기술 실현성:** 대전 교통 API를 실제 키로 호출해 페이지 수, 일일한도, 갱신주기, 레코드를 확인한다.
3. **P1 / 데이터 결합:** 공사구간을 링크·방향·기간에 매핑하고 신뢰등급과 보류 기준을 남긴다.
4. **P1 / 차별성:** 2024·2025 수상작과 개발 중 7개 신서비스를 원본 수준으로 대조한다.
5. **P1 / 사용자 타당성:** 교통대책 담당자와 배차담당자가 실제로 바꿀 수 있는 의사결정과 필요한 출력 형식을 검증한다.
6. **P1 / 모델 타당성:** 최소 수집기간의 한계를 공개하고 시간순 기준모델 비교를 수행한다.
7. **P2 / 연동:** 특정 업체 연동 대신 독립형 CSV·JSON 계약을 먼저 구현하고 오류·결측 규격을 확정한다.
